Virtual Jaguar v2.2.0 — HLE BIOS, save states, 22 games fixed

76 commits, single author. HLE BIOS that actually boots most commercial titles without a real BIOS image. Save states, SRAM/EEPROM, cheat codes, RetroAchievements. Audio rewrite, ~2x on the hot paths, SIMD blitter with bit-exactness CI. Closes #27, #85, and many sub-issues of #38.

Open Source Emulation Atari Jaguar libretro C/C++

What’s in this release

v2.2.0 closes a bunch of long-standing libretro tracker issues (#27, #85, sub-issues of #38) and adds modern emulator features the core didn’t have.

The notable bits: HLE BIOS compat improvements, 22 previously-broken games fixed, save states/rewind/RetroAchievements, SRAM/EEPROM persistence, cheat support, an audio pipeline rewrite, ~2x on the hot paths via a SIMD blitter, per-button remapping for the 21-key controller, pre-built cores for 14 platforms.

Nothing world-changing. Just a release that gets the core where users actually want it to be.

HLE BIOS: what changed

Virtual Jaguar has had an HLE BIOS path for a while; it’s the default for most users. What didn’t work cleanly before this release: a chunk of homebrew getting rejected by the cart-auth check, several games that probed DSP state at startup and got the wrong values, and a handful that depended on specific Tom/Jerry register init values the HLE wasn’t setting.

Quick legal note since people ask: the HLE path is not a reverse-engineered copy of the real BIOS. No disassembled BIOS code, no boot logo, no Atari-copyrighted asset. It re-implements the spec the BIOS implements so the hardware ends up in the same observable state by the time the game gets control. Same legal basis Saturn and PSP HLE BIOSes have used for years.

The fixes in v2.2.0:

  • Cart auth on homebrew. The Jaguar BIOS verifies a small signature region in the cart header. Not a cryptographic hash, just a sequence of expected bytes, but the BIOS will refuse to boot if it doesn’t see them. Licensed games all have valid signatures; homebrew sometimes doesn’t. The HLE replacement validates the same region but is more permissive on the homebrew path so legitimately-built indie ROMs (the 240p Test Suite port, for example) boot without rejection.
  • OS init coverage. Tom and Jerry have dozens of memory-mapped registers the BIOS pre-configures: clock dividers, DRAM refresh timing, the OP’s linked-list base pointer, the DSP’s default sample rate. The HLE was missing several. Games that assumed those values at entry would behave erratically. The HLE init now writes the same defaults in the same order as the real BIOS.
  • Audio init bit-exact. This one ate two days. The DSP has to land in an exact idle state (specific PC, specific register file, specific RAM contents at a couple of fixed addresses), or games that probe the DSP at startup fail their detection check and refuse to use audio. Got it bit-exact by diffing the post-boot DSP RAM dump from a real machine against the HLE-initialized version, byte by byte. The diff was non-zero in three places. Each one was a separate regression to track down.

Net: more games boot cleanly through HLE.

22 games that didn’t work before

Even the games that “worked” in older builds often had specific glitches: missing audio, broken sprite layers, hangs on level transitions. v2.2.0 fixes 22 titles that previously didn’t boot or didn’t run far enough to be playable. Four worth walking through:

  • Doom. The white whale. Doom on Jaguar is a tightly-tuned port that pushes the blitter harder than almost any other commercial game and is extremely sensitive to blitter-pipeline timing. The previous core had a one-cycle race in the blitter’s outer-loop dispatch that Doom would trip about 30 frames into the title screen; the symptom was a freeze with garbled VRAM. The fix was reordering two blitter-state transitions so the cycle accounting matches what the real chip does. Doom now boots and runs at full speed.
  • Syndicate. Polls the DSP’s status register every audio frame to decide whether to push a new sample buffer. The old core handled that poll lazily and would occasionally return a stale value when the DSP was mid-transition; Syndicate would mistake that for a hang and crash itself out of the audio thread. Tightening the DSP status-register read to be cycle-accurate fixed it.
  • Atari Karts. The Object Processor builds the Jaguar’s sprite layers from a linked list of OP descriptors. Atari Karts uses a non-standard descriptor ordering for the road-and-kart layers. The old OP walked the list in the wrong direction, drawing karts under the track surface. Re-read the OP datasheet, fixed the traversal.
  • Iron Soldier 2. Hung on the mission-select screen because of a missed interrupt during cart-bank switching. The bank-switch path now flushes pending interrupts before swapping, which matches real-hardware behavior.

The other 18 are a mix of audio fixes, OP fixes, m68k addressing-mode corner cases, and DSP timing improvements. Per-game status is in docs/emulation-bug-hunt-todos.md. If a title was broken for you before, try v2.2.0.

Audio: a 12-year-old bug, finally found

The frame-edge dropout bug has been in the core since at least 2014. Symptom: every few seconds, a brief click or half-sample of silence right at the audio-frame boundary. Not catastrophic but enough to make extended sessions unpleasant.

The diagnostic took longer than the fix. Built a bit-exactness test harness: capture the DSP’s sample output from a real Jaguar across a known scene from a known game, capture the same output from the emulator, diff sample-by-sample. The diff was almost always zero. The two waveforms were sample-identical. Except at frame boundaries, where the emulator was occasionally emitting a sample that had been written half a cycle late. The DSP scheduler was draining its sample buffer just before the audio mixer pulled, instead of just after.

The fix was a one-line ordering change in the per-frame schedule. retro_run() now does update_input -> DACPrepareFrame -> JaguarExecuteNew -> cheat_apply_all -> SoundCallback -> video_cb, so the audio callback sees the same JERRY state the frame was rendered against and doesn’t drift. Validation took a week: ran the harness across 31 games, each across a 60-second scene, zero mismatches anywhere. The harness is now part of CI; a regression in the audio path can’t merge silently.

Save states, rewind, RA

None of these existed in the previous core. Each was harder than it sounds.

Save states. Conceptually simple: serialize the entire machine state to a blob, deserialize later. The Jaguar is unusually heterogeneous though. Six chips hold state: the m68k CPU, the GPU (RISC core), the DSP (another RISC core, different ISA), the Object Processor, the Blitter, and the audio subsystem. Each had its own ad-hoc representation of state, and several held pointers into shared memory that didn’t serialize cleanly. The work was making each chip’s state deterministically serializable: stable byte layout, no embedded pointers, no undefined fields. With that done, saving and loading is the obvious thing. Run-ahead works for free.

Rewind rides on save states with a circular buffer. Every 4 frames the emulator captures a state, ring-buffers it, and on rewind walks the ring backwards. Buffer is sized for ~30 seconds of rewind by default. Cost is mostly memory; CPU overhead is negligible because the per-frame capture is well under a millisecond on a modern machine.

RetroAchievements. Jaguar has been one of the few major retro consoles with no achievement support. Users in the RA community had no Jaguar sets to work on, no way to track progress, and the platform was effectively invisible to that whole sub-community. v2.2.0 wires up the standard RA hooks (RC_CONSOLE_ATARI_JAGUAR, memory-map exposure, frame-tick callbacks, hardcore-mode enforcement). ~20 fresh achievement sets are being authored on RA’s side as I write this.

SRAM and EEPROM persistence got fixed in the same release. Save data now survives soft resets, which it didn’t before for cartridges that used the on-cart EEPROM (a small but loud subset of the library, mostly the racing titles).

21-key remap

The Jaguar controller has a digital d-pad, three face buttons (A, B, C), Pause, Option, and a 12-key numeric keypad. The 1993 design assumed games would ship with paper overlays you’d snap on to label what each keypad key does in that specific game.

The problem for emulation is that no modern controller has 21 inputs, and the keypad keys are load-bearing for a lot of titles. Alien vs. Predator uses the keypad for door codes and inventory; Iron Soldier uses it for weapon selection; the homebrew scene uses it constantly. Dropping the keypad onto “keyboard only” isn’t acceptable on platforms like iOS where there is no keyboard.

v2.2.0 ships per-button remapping for the entire 21-key layout via virtualjaguar_p1/p2_retropad_* core options. Every keypad key is independently assignable to any input the libretro frontend exposes: analog-stick directions, modifier-button combos, on-screen overlays, whatever your frontend supports. The defaults are sensible for the most-used titles (keypad mapped to a virtual on-screen overlay on touch platforms; combo-buttons on traditional controllers). The override path is one menu deep. Frontend authors can ship per-game defaults; users can override per-game.

~2x on hot paths

I wasn’t chasing performance in this release. I was chasing correctness. Several of the correctness fixes turned out to also be much faster than what they replaced. The headline is roughly 2x on the hot paths, profile-driven and measured against v2.1 on the same hardware running the same scenes.

The biggest win was the blitter. The old implementation was a giant interpreter loop that re-decoded the blitter’s mode register on every operation. The rewrite hoists the mode decoding out of the inner loop, pre-computes a small dispatch table, and uses SIMD for the actual pixel-fill stages where the blitter is moving large rectangular spans. SSE2 on x86, NEON on ARM, scalar fallback otherwise. The bit-exactness harness covers the blitter rewrite too: pixel-perfect output across a battery of test scenes, divergence between SIMD and scalar paths fails CI immediately.

Other wins, less dramatic: tightened the m68k decoder’s common-case dispatch, replaced a few memcpy calls with structured assignment in the OP’s descriptor walk, inlined a couple of accessor functions the compiler wasn’t consistently inlining. Plain C where it helps; SIMD where it really helps.

The bit-exactness harness made all of this safe. Without it, a SIMD blitter rewrite is the kind of change you’re scared to merge because regressions are subtle and only visible in specific games on specific scenes. With pixel-perfect comparison against a known-good baseline, the regression surface collapses to “does the harness still pass?”

What’s next

Jaguar CD. The Atari Jaguar CD add-on shipped a small but interesting library, including some of the most graphically ambitious Jaguar software made. Emulation of it has been historically poor. Today, the only meaningful Jaguar CD emulation outside MiSTer FPGA is fragmentary. Bringing it into virtualjaguar-libretro is the logical next step, and v2.2.0’s improved core makes that work substantially easier. PR #119 stacks the IRQ/HLE/audio fixes that #109 was working around; once it lands, the CD work in #109 reduces from +426 lines to +22.

On the distribution side: Provenance ships virtualjaguar-libretro as one of its Jaguar cores, so users on iOS, tvOS, and macOS will pick up all of v2.2.0’s improvements automatically in the next core update. No re-downloading, no BIOS sourcing, no setup. The HLE BIOS path is especially nice on iOS, where filesystem access for BIOS files is awkward at best.

Related work in the JoeMatt/atari_jaguar_240p_test_suite repo is one of the few things actively producing new homebrew Jaguar ROMs. That makes it one of the best stress-tests for the core: every release of the test suite immediately exercises virtualjaguar-libretro’s edge cases. The two projects feed each other.

If you find a regression, file an issue. Single-author releases are fast, but they also have single-author blind spots. The bit-exactness harness catches a lot. It doesn’t catch everything. The compatibility table in the wiki is the best place to confirm whether a specific title is verified; PRs against it are welcome from anyone who can run a few minutes of a game and report what they see.
