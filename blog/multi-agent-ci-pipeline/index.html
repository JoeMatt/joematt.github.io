The Multi-Agent CI Pipeline I Built So I Could Ship Open Source Like a Team of One

3 agents. 9 workflow files. One shared dispatch vocabulary. Reviewer-model auto-selection by static diff analysis — and a scheduled poller that bypasses GitHub’s undocumented bot-actor approval gate. Here’s how the orchestration layer behind Provenance EMU works, and why the routing matters more than the model.

AI engineering Open source GitHub Actions Multi-agent Claude Cursor Kimi

01Where the wall actually is

I maintain Provenance EMU. The numbers, since you’re going to ask: 6,300 GitHub stars, 79 contributors, 388,000+ release downloads, an App Store launch in January 2025, and an active Discord with 9,000+ members. I do this on top of a full-time job at Wayfair on the App Infrastructure team, plus three US patents I picked up doing location analytics in another life. I’m not telling you this to brag — I’m telling you because the rest of the post only makes sense if you understand what scale of project this is for one person.

The honest story about solo open-source maintainers is that we all hit a velocity wall around year five. Mine showed up on schedule. PRs pile up faster than you can review them. Localization rots because nobody (including you) wants to manually re-translate strings every sprint. Documentation drifts. Wiki spell-check breaks and stays broken because fixing it takes ten minutes you don’t have. Code review — the actual technical evaluation, not the rubber-stamping — becomes the thing that gates everything else.

The lazy answer is “use Copilot more.” The actual answer, for me, was building an orchestration layer. Not an agent — a layer that routes agents. Three of them, sharing one dispatch vocabulary, with a model selector that picks the reviewer based on what the diff actually contains. Nine workflow files in .github/workflows, all public, all running in production today. The TL;DR: my PR throughput is up significantly over the last two quarters, my median review latency dropped from days to hours, and my Anthropic spend across the entire org is around $80 a month. None of this required a custom agent framework. It required reading the GitHub Actions docs carefully and being willing to write the boring routing code.

Three things to notice. First, the router is dumb on purpose — it’s just if statements over branch names, labels, and a handful of grep patterns. Second, all three agents (Claude Code, Cursor Agent, Kimi via the Anthropic-compatible endpoint) consume the same workflow_dispatch inputs. Third, Copilot does the review pass, but a cron job — not the review event itself — is what fires the next stage. That last detail is the one that took me longest to figure out, and we’ll get to it.

03Reviewer model auto-selection by static diff analysis

This is the part I’m most proud of, and it’s also the cheapest piece of code in the whole system. Maybe twelve lines of bash.

The problem: Claude Sonnet (4.5, currently) is fast and inexpensive — you can run it all day on small reviews and barely move the needle on your bill. Claude Opus is dramatically smarter on hard problems, but the per-token cost is something like an order of magnitude higher. If you pick by gut, you waste money on UI-only diffs that Sonnet would handle perfectly, or you bottleneck reviews of genuinely tricky code by routing it to a model that doesn’t have the headroom. Most teams I’ve seen either default to whichever is in vogue (currently Opus, which is expensive) or default to whichever is cheap (Sonnet, which misses things).

The solution: stop trying to be clever and just grep the diff. There’s a small dictionary of patterns that, in my codebase, reliably indicate “this is the kind of change where a smarter model pays for itself.” They’re all about thread safety, persistence boundaries, and Swift concurrency:

# inside the router workflow
DIFF=$(gh pr diff "$PR_NUMBER")

# patterns where Opus consistently beats Sonnet in my codebase:
#  - Realm: thread-confined object model, easy to corrupt
#  - @Model: SwiftData, similar story
#  - actor / @MainActor: Swift concurrency boundaries
#  - @synchronized / NSLock: explicit locks; subtle deadlocks
#  - DispatchQueue: GCD; race conditions are non-obvious
#  - C++ files: the emulation core has real concurrency

if echo "$DIFF" | grep -qE 'Realm|@Model|@synchronized|^\s*actor\s|@MainActor|DispatchQueue|NSLock|\.cpp$|\.mm$'; then
    echo "model=claude-opus-4-5" >> "$GITHUB_OUTPUT"
    echo "reason=high-risk-pattern-detected" >> "$GITHUB_OUTPUT"
else
    echo "model=claude-sonnet-4-5" >> "$GITHUB_OUTPUT"
    echo "reason=default" >> "$GITHUB_OUTPUT"
fi

That’s the whole heuristic. The reason output gets posted as a PR comment so I can audit it later, which has been useful: a few times I’ve added new patterns to the dictionary after a Sonnet review missed something I felt Opus would have caught. The dictionary is a living thing.

Why this works is, I think, the interesting part. Thread-safety bugs and persistence-boundary bugs are exactly the class of problem that a smarter model pays for — they require holding a lot of context about the surrounding code, the lifecycle of an object, and which queue the current method runs on. UI tweaks, copy changes, and dependency bumps don’t require any of that. The diff itself contains enough signal to predict, statically, which class of review you’re about to do. You don’t need a router model. You don’t need an LLM in the loop at all to make this routing decision. grep is fine.

The generalizable pattern: build your own diff-pattern dictionary based on what your codebase actually breaks on. If you’re a Rails shop, your patterns might be after_commit, has_many :through, anything touching ActiveRecord::Base.transaction. If you’re a Go shop, it’s go func, channel ops, anything in the sync package. Look at your last twenty production incidents and ask: was the diff that caused this distinguishable, in advance, by file extension or grep pattern? In my experience the answer is almost always yes.

The quiet payoff. Roughly 70% of my PRs get the Sonnet path; 30% get Opus. Before this routing existed, I was paying Opus prices for everything “to be safe.” The bill is now about a third of what it was, and I haven’t had a Sonnet review miss something Opus caught in production yet. That last sentence is true today; ask me in six months.

04Three agents, one action vocabulary

The annoying truth about most multi-agent setups I’ve seen is that teams build a separate workflow tree per agent, and end up maintaining N copies of basically the same logic. There’s a Claude workflow that does a thing, a Copilot workflow that does the same thing slightly differently, and a Cursor workflow that does the same thing yet again, and now any change to the orchestration logic has to be made three times.

The better pattern — the one I landed on after reverting two earlier iterations — is: one shared dispatch vocabulary, agents are workers. The orchestration layer doesn’t know which agent it’s talking to. It just knows which action it wants performed. Routing to a specific agent is a separate concern that happens at the worker level.

The vocabulary, currently four verbs:

# every agent worker accepts these workflow_dispatch inputs
inputs:
  action:
    type: choice
    options:
      - implement_issue       # take an issue, open a PR
      - fix_ai_review         # respond to a Copilot review with fixes
      - fix_rebase_conflict   # resolve merge conflicts on a stale branch
      - ai_approved           # signal review complete; ready for human gate
  issue_number:
    type: string
  pr_number:
    type: string
  model:
    type: string
    default: claude-sonnet-4-5

Routing happens by branch suffix and label. A branch named fix/issue-1234-cursor goes to the Cursor lane. A branch with no suffix and no special label goes to Claude. A label called ai:fallback-kimi forces the Kimi worker. The router workflow does that mapping in — again — about ten lines of bash. Adding a fourth agent (I’ve been eyeing Aider and Devin’s API) is a copy-paste of the worker workflow plus one new branch suffix in the router.

Why this matters in practice: when I want to change the dispatch vocabulary — say, add a fifth verb for fix_localization_drift — I touch one input definition, propagate the new case to each worker, and the orchestration layer above doesn’t care. The agents are interchangeable. That’s the whole goal of an orchestration layer, and most teams accidentally build the opposite.

05Kimi as Anthropic-API-compatible fallback

Claude Code is great. It’s also a binary distributed by Anthropic that hardcodes a lot of Anthropic-SDK behavior. So how do you fail over when your credits run out, or when Anthropic has a partial outage, without rewriting your worker workflow?

It turns out Moonshot’s Kimi exposes an Anthropic-compatible coding endpoint specifically for this case. Same API surface, different model behind it. The fallback is a single environment variable:

- name: Run Claude Code (fallback to Kimi K2)
  env:
    ANTHROPIC_API_KEY: ${{ secrets.KIMI_API_KEY }}
    ANTHROPIC_BASE_URL: https://api.kimi.com/coding
    # ANTHROPIC_MODEL is set in the router; for Kimi we use kimi-k2
  run: |
    claude --print "..." --model kimi-k2

The same Claude Code binary, with no other workflow changes, now talks to Kimi. The trigger in production is rate-limit / credit-exhaustion errors from the Anthropic endpoint — those bubble up to the workflow as a non-zero exit, the next attempt sets the Kimi env vars, and the work continues. I’ve had this fire maybe four or five times in the last quarter. Each time, the PR finished without me noticing until I looked at the workflow log.

The reason this is worth writing about: most multi-agent setups assume the agents have wildly different SDKs and design around that assumption. In practice, OpenAI-compatible and Anthropic-compatible endpoints have become the lingua franca, and you can swap models behind your existing tooling for the price of a base URL.

06The scheduled poller that bypasses GitHub’s bot-actor gate

This is the second piece I’m proud of, and it’s the one nobody talks about. If you’ve never tried to chain a GitHub bot’s output into another workflow, you’re about to find out why.

The setup: when Copilot finishes reviewing a PR, I want my Claude worker to read that review and address the feedback automatically. The intuitive way to wire this is with a pull_request_review event — GitHub fires it whenever a review is submitted. Easy.

Except: GitHub silently blocks event-triggered workflows when the actor is a bot — specifically copilot[bot], github-actions[bot], and friends. They get gated behind an action_required approval that’s only visible deep in the UI, and the workflow doesn’t fire until a human clicks an approve button. This is a security policy, and it’s a sensible one in the abstract. But it makes bot-to-bot chaining impossible by default. The downstream workflow simply never fires. There’s no error. It’s just stuck.

I tried the obvious workarounds first. pull_request_target doesn’t help — the actor gate still applies. Manually approving each one defeats the point of the automation. Service accounts get re-flagged as bots after enough activity. None of these scale.

The workaround that actually works: a scheduled cron job, running every ten minutes, that polls for new Copilot reviews and dispatches the next workflow itself. The scheduled run’s actor is GitHub Actions, not Copilot, so the bot-actor gate doesn’t apply. The downstream workflow fires normally.

name: Poll Copilot reviews
on:
  schedule:
    - cron: '*/10 * * * *'    # every 10 min
  workflow_dispatch:           # manual kick

# WHY THIS EXISTS, READ THIS BEFORE DELETING:
# GitHub silently gates event-triggered workflows when the actor is
# copilot[bot] (action_required approval). pull_request_review
# events from Copilot do NOT fire downstream workflows by default.
# A scheduled run, however, has the github-actions[bot] actor and
# is NOT subject to that gate. So we poll instead of subscribing.

permissions:
  pull-requests: write
  contents: read
  actions: write

jobs:
  poll:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Find PRs with new Copilot reviews
        env:
          GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        run: |
          # find PRs reviewed by copilot[bot] in the last ~15 min
          # that don't yet have an "ai:reviewed" label
          gh pr list --json number,labels,reviews \
            --jq '.[] | select(.reviews[]? | select(.author.login == "copilot[bot]"))' \
          | while read -r pr; do
              # dispatch the fix_ai_review action — this run is
              # the trusted scheduled actor, so no gate
              gh workflow run claude-worker.yml \
                -f action=fix_ai_review \
                -f pr_number="$(echo "$pr" | jq -r .number)"
            done

Ten-minute latency is fine for my use case — I’m not trying to optimize for sub-second turnarounds, I’m trying to keep the loop closed without me in it. The poller is the bridge between “Copilot finished reviewing” and “Claude starts addressing the feedback,” and it’s the missing piece in every multi-agent CI tutorial I’ve read. If you take one thing from this post, take this: scheduled actors bypass the bot-actor gate, and that’s the only reliable way to chain bot output into another workflow on GitHub today. I’ve filed feedback with GitHub on this; I don’t expect it to change soon, because the gate is doing its job.

07Self-healing wiki CI

The Provenance wiki lives in a separate repo and has its own CI — spell check, link check, structural validation. It used to break every couple of weeks because someone (often me) merged a doc edit with a typo or a dead link, and the failure would sit there until I noticed.

It now self-heals. The CI failure triggers Claude via the official anthropics/claude-code-action, Claude opens a PR with the spell-check or link-check fixes, and — this is the part that delights me — requests Copilot review on its own PR. The poller from section 06 then picks up the Copilot review, and if Copilot approves, the PR auto-merges via a branch-protection bypass token. Loop closes without me.

Hit rate: roughly 80% of wiki CI failures self-resolve within an hour. The other 20% are real problems — broken external links, structural changes the doc tooling doesn’t handle — and they bubble up to me via a Slack notification. The signal-to-noise on my notifications is finally what I want.

08Diff-aware Claude Haiku localization

Provenance ships with strings localized into eleven target languages. The naive approach to keeping those in sync — the one I had for years — is: when you change en.lproj/Localizable.strings, re-translate the entire file to every target language. That’s expensive, slow, and pointlessly destructive of any human edits localizers may have made downstream.

The better pattern is diff-aware. On every push that touches the English strings file, a workflow does:

  1. Diff en.lproj/Localizable.strings between HEAD~1 and HEAD
  2. Extract the keys that are new or whose values changed
  3. Send only those keys to Claude Haiku with a translation prompt and the existing target-language file as context
  4. Apply Haiku’s output as a patch to each of the eleven target files
  5. Open a PR labeled ai:localization for human spot-check

Haiku is the right model here for two reasons: it’s cheap enough to run on every push without thinking about it, and translation of short UI strings is the kind of task where Sonnet’s extra capability buys you nothing measurable. The cost of a full-sweep localization run is now in the single-digit cents instead of the dollars-per-language it used to be, and the latency is under a minute. The pattern reuses cleanly for any string-keyed localization pipeline — not just iOS .strings; same shape works for Android strings.xml, gettext .po, JSON i18n bundles, anything with stable keys.

09Real outcomes, honestly

I’ve been running this stack in roughly its current form for about nine weeks. Some honest numbers and observations:

  • PR throughput is up significantly. I’m being deliberately vague because the comparison is unfair — before this stack, I was the bottleneck on every PR; after it, I’m the bottleneck on the interesting PRs. Median time-to-first-review dropped from about 36 hours to under 2 hours.
  • Total Anthropic spend across the org: ~$80/month. That covers the routing, the wiki self-heal, the localization pipeline, the Claude Code workers themselves, and the occasional Opus reviewer. The routing-by-diff-grep is the single biggest cost lever.
  • What didn’t work: I tried, briefly, per-PR Cursor agent assignment based on file extension — e.g. all .swift files go to Cursor, all .cpp go to Claude. Abandoned within two weeks. The file-type signal turned out to be a poor proxy for which agent does better; the diff-pattern signal (section 03) is much stronger.
  • What surprised me: the AI reviewers catch real bugs more often than I expected, especially around threading and Realm transactions. They also occasionally catch documentation drift (a function’s doc-comment no longer matches its signature), which I never saw coming. The false-positive rate is non-trivial — maybe 15–20% of comments are noise — but a quick triage pass is much cheaper than missing a real bug.
  • What I still don’t trust them with: design decisions, public API changes, anything in the C++ emulation core that touches sync timing, anything that affects file format compatibility with prior releases. Those PRs get the AI review for completeness, but the merge decision is mine and only mine.

10The takeaway

Multi-agent CI isn’t about replacing engineers. It’s about removing the routine review-and-triage layer that, for solo maintainers and small teams, is the actual rate-limiting step. The point of the orchestration isn’t to make the agents smarter — they’re already smart enough — it’s to put the right agent on the right work, automatically, without a human standing in the middle of the routing decision.

If you take three things from this post: (1) route by static diff analysis, not by gut and not by router-LLM; (2) build one shared dispatch vocabulary and treat agents as interchangeable workers behind it; (3) use a scheduled poller for any chain that crosses the GitHub bot-actor gate, because subscribing to the event will silently fail and you’ll spend a weekend figuring out why.

All nine workflow files behind this are public. They live in the various Provenance repos under .github/workflows. Copy them. Fork them. Adapt the dispatch vocabulary to your stack. The orchestration is the part worth stealing; whichever agent you put behind it is replaceable.
