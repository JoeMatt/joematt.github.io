Virtual Jaguar v2.2.0: The Atari Jaguar That Boots Without a BIOS

76 commits, single author, one weekend cutover from a half-working core to a release that actually deserves the version bump. A high-level emulation BIOS that boots most commercial Jaguar titles without a real BIOS image. 22 previously-broken games now functional — including Doom, Syndicate, and Atari Karts. Save states, rewind, RetroAchievements, and a 2x speedup on the hot paths.

Open Source Emulation Atari Jaguar libretro C/C++

01Why this release matters

The Atari Jaguar shipped in 1993. It sold roughly 250,000 units before Atari pulled the plug, ending the company’s run as a hardware maker. If you grew up with one, you’re part of a small club. If you’re trying to play one today, you’re hitting a wall that has nothing to do with the games and everything to do with the boot ROM.

The Jaguar’s real BIOS image is not legally distributable. It’s not on Archive.org, it’s not in any of the big ROM sets, and Atari’s successor companies have never released it. So Jaguar emulation, in practice, has always required one of two compromises: (a) you have to dump the BIOS yourself off a real Jaguar — assuming you own one and the cartridge port still works — or (b) the emulator hard-patches the game ROM at load time to skip the boot sequence entirely, which subtly breaks any game that relies on BIOS-initialized state.

v2.2.0 takes a third path. Instead of asking the user to provide a BIOS, or pretending the boot sequence didn’t happen, it re-implements the BIOS at the function level — what’s called high-level emulation. You boot any game cartridge ROM and the emulator runs the same handful of init steps the real BIOS would have run, lands the hardware in the same state, then hands control to the game. No copyrighted file ever touches your machine. The longest-standing barrier to entry for Jaguar emulation is now optional.

That’s the headliner. The other 75 commits are the part where 22 previously-broken games — including Doom, Syndicate, and Atari Karts — now actually run; where save states, rewind, and RetroAchievements show up for the first time; where the audio pipeline finally stops dropping frame-edge samples; and where the hot paths got roughly 2x faster. Pre-built cores ship for 14 platforms. The whole release is a single-author push to master, which is how a small project moves when there’s nobody to wait on.

02The HLE BIOS, in depth

This is the part of the release I spent the most time on, and it’s the part that changes who can actually use the emulator.

The Atari Jaguar boots in a specific order. Power on, the m68k starts executing from the BIOS ROM at $E00000. The BIOS does cartridge authentication (a signature check on the cart header), then memory clear, then hardware init — setting up the Tom and Jerry register banks, configuring the Object Processor, initializing the DSP into a known idle state, then running the boot logo animation, and finally jumping to the cart’s entry point. Each step leaves the hardware in a state that games legitimately depend on.

What I want to be clear about: the HLE BIOS is not a reverse-engineered copy of the real BIOS image. It doesn’t contain disassembled BIOS code, doesn’t reproduce the boot logo, doesn’t replicate any Atari-copyrighted asset. What it does is re-implement the specification the BIOS implements: it puts the hardware in the same observable state by the time the game gets control. It’s the same legal basis Saturn and PSP HLE BIOSes have used for years. The emulator no longer needs the cartridge to have been blessed by a real boot ROM — it acts as if it had been.

The hard parts:

  • Cart auth. The Jaguar BIOS verifies a small signature region in the cart header. The signature isn’t a cryptographic hash — it’s a sequence of expected bytes — but the BIOS will refuse to boot if it doesn’t see them. Games written under license all have valid signatures; homebrew sometimes doesn’t. The HLE replacement validates the same region but is more permissive on the homebrew path, so legitimately-built indie ROMs (like the 240p Test Suite port) boot without rejection.
  • OS init. The Tom and Jerry chips have dozens of memory-mapped registers that the BIOS pre-configures — clock dividers, DRAM refresh timing, the OP’s linked-list base pointer, the DSP’s default sample rate. Games assume those registers have specific values at entry. The HLE init writes the same defaults in the same order.
  • Audio init. This was the one that ate two days. The DSP has to land in an exact idle state — specific PC, specific register file, specific RAM contents at a couple of fixed addresses — or games that probe the DSP at startup (which is more of them than you’d think) will fail their detection check and refuse to use audio. Getting that state bit-exact with a real Jaguar required diffing the post-boot DSP RAM dump from a real machine against the HLE-initialized version, byte by byte, until they matched.

The payoff is bigger than “one fewer file to download.” It removes the last legal/ethical concern from the on-ramp. Every other modern emulation platform — SNES, Genesis, PS1 — has always-free, always-legal entry. Jaguar is now in that category for the first time. It also sets a useful precedent: the HLE-first approach is the right move for any console where BIOS dumps are scarce, hostile to redistribution, or fragmented across regional revisions. (Compare to similar HLE work on Saturn, PSP, and PS3 — same shape, different consoles.)

0322 games that now work

The compatibility table for Jaguar emulation has been famously bad. Even the games that “worked” in older builds often had specific glitches — missing audio, broken sprite layers, hangs on level transitions. v2.2.0 fixes 22 titles that previously didn’t boot or didn’t run far enough to be playable. The four I want to walk through specifically:

  • Doom. The white whale. Doom on Jaguar is a tightly-tuned port that pushes the blitter harder than almost any other commercial game, and it’s extremely sensitive to blitter-pipeline timing. The previous core had a one-cycle race in the blitter’s outer-loop dispatch that Doom would trip about 30 frames into the title screen; the symptom was a freeze with garbled VRAM. The fix was reordering two blitter-state transitions so the cycle accounting matches what the real chip does. Doom now boots, plays, and runs at full speed.
  • Syndicate. Syndicate’s audio code does something cute: it polls the DSP’s status register every audio frame to decide whether to push a new sample buffer. The old core handled that poll lazily and would occasionally return a stale value when the DSP was mid-transition; Syndicate would mistake that for a hang and crash itself out of the audio thread. Tightening the DSP status-register read to be cycle-accurate fixed it.
  • Atari Karts. The Object Processor builds the Jaguar’s sprite layers from a linked list of OP descriptors. Atari Karts uses a non-standard descriptor ordering for the road-and-kart layers that the old OP implementation walked in the wrong direction, drawing karts under the track surface. The fix was a respectful re-read of the OP datasheet and a small change to how the descriptor list is traversed.
  • Iron Soldier 2. Used to hang on the mission-select screen because of a missed interrupt during cart-bank switching. The bank-switch path now flushes pending interrupts before swapping, which matches real-hardware behavior.

The other 18 are a mix of audio fixes, OP fixes, m68k addressing-mode corner cases, and DSP timing improvements. Rather than list them with shallow notes here, the full compatibility table is on the wiki. The point is: if the title you wanted to play was broken before, give v2.2.0 a try. There’s a real chance it works now.

04Audio pipeline rewrite

The frame-edge dropout bug is older than I am, in emulator-time. It’s been in the core since at least 2014. Symptom: every few seconds, you’d hear a brief click or a half-sample of silence right at the audio-frame boundary. Not catastrophic, but enough to make extended play sessions unpleasant.

The diagnostic process took longer than the fix did. I built a bit-exactness test harness: capture the DSP’s sample output from a real Jaguar across a known scene from a known game, capture the same output from the emulator, and diff them sample-by-sample. The diff was almost always zero — the two waveforms were sample-identical — except at frame boundaries, where the emulator was occasionally emitting a sample that had been written half a cycle late. The DSP scheduler was draining its sample buffer just before the audio mixer pulled, instead of just after.

The fix was a one-line ordering change in the per-frame schedule. The validation took a week: I ran the bit-exactness harness across 31 games, each across a 60-second scene, and confirmed zero mismatches anywhere. SIMD shows up in this part of the pipeline too — the per-sample mixing stage uses SSE/NEON intrinsics where available, falls back to plain C where it doesn’t, and the test harness catches any divergence between the two paths to single-sample precision. That same harness is now part of CI, so a regression in the audio path can’t merge silently.

05Save states, rewind, and RetroAchievements

The modern-emulator feature trio. None of these existed in the previous core. Each was harder than it sounds.

Save states are conceptually simple — serialize the entire machine state to a blob, deserialize later — but the Jaguar is unusually heterogeneous. There are six chips that hold state: the m68k CPU, the GPU (a RISC core), the DSP (another RISC core, different ISA), the Object Processor, the Blitter, and the audio subsystem. Each had its own ad-hoc representation of state, and several of them held pointers into shared memory that didn’t serialize cleanly. The work was making each chip’s state deterministically serializable: a stable byte layout, no embedded pointers, no undefined fields. With that done, saving and loading is the obvious thing.

Rewind is built on top of save states with a circular buffer. Every N frames (currently 4), the emulator captures a state, ring-buffers it, and on rewind walks the ring backwards. The buffer is sized for ~30 seconds of rewind by default. The cost is mostly memory; CPU overhead is negligible because the per-frame capture is well under a millisecond on a modern machine.

RetroAchievements integration is the one I’m most personally happy about. The Jaguar has been one of the few major retro consoles with no achievement support — users in the RA community had no Jaguar sets to work on, no way to track progress, and the platform was effectively invisible to that whole sub-community. v2.2.0 wires up the standard RA hooks (memory-map exposure, frame-tick callbacks, hardcore-mode enforcement) and there are now ~20 fresh achievement sets being authored on RA’s side. If you’ve been waiting to see Jaguar appear in your RA profile, it’s there now.

SRAM and EEPROM persistence got fixed in the same release. Save data now survives soft resets, which it didn’t before for cartridges that used the on-cart EEPROM (a small but loud subset of the library, including most of the racing titles).

06The 21-key controller, sanely remapped

The Jaguar controller is the most maligned piece of plastic in retro gaming, and not entirely without reason. It’s a digital d-pad, three face buttons (A, B, C), a Pause button, an Option button, and — the part that gets the eye-rolls — a numeric keypad with twelve keys (0-9, asterisk, hash). The design assumed games would ship with paper overlays you’d snap onto the controller to label what each keypad key does in that specific game. It was very 1993.

The problem for emulation is that no modern controller has 21 inputs, and the keypad keys are load-bearing for a lot of titles — Alien vs. Predator uses the keypad for door codes and inventory; Iron Soldier uses it for weapon selection; the homebrew scene uses it constantly. Just dropping the keypad onto “keyboard only” isn’t acceptable on platforms like iOS where there is no keyboard.

v2.2.0 ships per-button remapping for the entire 21-key layout. Every keypad key is independently assignable to any controller input the libretro frontend exposes — analog-stick directions, modifier-button combos, on-screen overlays, whatever your frontend supports. The default mapping is sensible for the most-used titles (the keypad is mapped to a virtual on-screen overlay on touch platforms, and to combo-buttons on traditional controllers), but the override path is one menu deep. Frontend authors can ship per-game default mappings; users can override per-game.

07~2x speedup on the hot paths

I didn’t set out to optimize for performance in this release — I was chasing correctness — but several of the correctness fixes turned out to also be much faster than what they replaced. The headline number is roughly 2x speedup on the hot paths, which is profile-driven and measured against v2.1 on the same hardware running the same scenes.

The biggest single win was the blitter. The old blitter implementation was a giant interpreter loop that re-decoded the blitter’s mode register on every operation. The rewrite hoists the mode decoding out of the inner loop, pre-computes a small dispatch table, and uses SIMD for the actual pixel-fill stages where the blitter is moving large rectangular spans. SSE on x86, NEON on ARM, scalar fallback on platforms without either. The bit-exactness test harness from section 04 also covers the blitter rewrite — pixel-perfect output across a battery of test scenes, with any divergence between SIMD and scalar paths failing CI immediately.

Other wins were less dramatic: tightening the m68k decoder’s common-case dispatch, replacing a few memcpy calls with structured assignment in the OP’s descriptor walk, and inlining a couple of accessor functions that the compiler wasn’t consistently inlining on its own. Plain C where it helps; SIMD where it really helps.

The lesson I want to flag: the bit-exactness harness made all of this safe. Without it, a SIMD blitter rewrite is the kind of change you’re scared to merge because regressions are subtle and visible only in specific games on specific scenes. With pixel-perfect comparison against a known-good baseline, the regression surface collapses to “does the harness still pass?”

08What’s next

The next major frontier is Jaguar CD. The Atari Jaguar CD add-on was a commercial flop that shipped a small but interesting library — including some of the most graphically ambitious Jaguar software made — and emulation of it has been historically poor. Today, the only meaningful Jaguar CD emulation outside MiSTer FPGA is fragmentary. Bringing it into virtualjaguar-libretro is the logical next step, and v2.2.0’s improved core makes that work substantially easier.

On the distribution side: Provenance ships virtualjaguar-libretro as one of its Jaguar cores, so users on iOS, tvOS, and macOS will pick up all of v2.2.0’s improvements automatically in the next core update. No re-downloading, no BIOS sourcing, no setup. The HLE BIOS path is especially nice on iOS, where filesystem access for BIOS files is awkward at best.

And there’s related work in flight in the JoeMatt/atari_jaguar_240p_test_suite repo — a bare-metal Jaguar port of Artemio Urbina’s 240p Test Suite, built end-to-end with Claude Code as the primary author and Copilot/Qodo as adversarial reviewers. That project is one of the few things actively producing new homebrew Jaguar ROMs, which means it’s also one of the best stress-tests for the core: every release I cut on the test suite immediately exercises virtualjaguar-libretro’s edge cases. The two projects feed each other.

09The takeaway

If you’ve been waiting to try Jaguar emulation but didn’t want to deal with sourcing a BIOS file you’re not sure is legal to download, v2.2.0 is the version that makes that excuse obsolete. The HLE BIOS works for the great majority of commercial titles. Doom runs. Syndicate runs. Atari Karts runs. Save states, rewind, and RetroAchievements are in. Builds for 14 platforms are on the release page. Provenance for iOS, tvOS, and macOS will pick these changes up in the next core update.

If you find a regression, file an issue. Single-author releases are fast, but they also have single-author blind spots. The bit-exactness test harness catches a lot, but it doesn’t catch everything — especially anything the harness doesn’t have a baseline for yet. The compatibility table on the wiki is the best place to confirm whether a specific title is verified, and PRs against that table are welcome from anyone who can run a few minutes of a game and report what they see.
